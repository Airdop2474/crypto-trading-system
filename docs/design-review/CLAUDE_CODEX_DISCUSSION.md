## 核心分歧：价格行为策略

### Codex 的观点

**建议：完全移除或远期规划**

**理由：**
1. 主观性太强，验收标准难定义
2. 容易引入 AI 过拟合
3. 会拉长第一个可验证闭环
4. 形态识别准确率难保证
5. 不如先把网格策略做扎实

**推荐路线：**
```
Phase 2A: 网格策略
Phase 2B: 网格 + 趋势过滤
Phase 2C: 网格 + 波动率过滤
Phase 2D: 网格 + 仓位管理
长期（Phase 8+）：再考虑价格行为
```

### Claude Code 的观点

**建议：后移到 Phase 5，但大幅简化**

**理由：**
1. **技术学习价值**
   - 网格策略纯数学，学不到市场结构
   - 价格行为需要理解支撑阻力、形态
   - 这是后续机器学习的基础

2. **验证系统扩展性**
   - 只有一个策略，无法验证策略管理框架
   - 无法测试策略切换逻辑
   - 无法验证多策略共存

3. **AI 价值验证**
   - 网格策略几乎不需要 AI
   - 价格行为天然适合模式识别
   - 可以真正验证 AI 在交易中的作用

4. **风险可控**
   - 后移到 Phase 5（网格稳定后）
   - 只回测，不进 Paper Trading
   - 不作为实盘策略

**折衷方案：**
```
Phase 3: 网格策略验证（完整）
Phase 4: Paper Trading（只网格，60天）
Phase 5: 价格行为策略（仅回测）⭐争议点
  - 缩短到 4周（不是8周）
  - 只实现核心形态（Pin Bar, Engulfing）
  - 只回测验证，不进模拟盘
  - 作为技术储备，不是主线
Phase 6: 风控强化
Phase 7: 实盘（只用网格）
```

## 为什么不建议 Phase 5 做价格行为

Claude Code 的折衷方案已经比原计划稳健，但我仍认为 Phase 5 做价格行为会有几个问题。

### 1. Paper Trading 后插入新策略会打断实盘准备

如果 Phase 4 已经完成 60 天 Paper Trading，下一步最自然的是强化风控、整理门禁、进入小资金实盘。

此时插入 4 周价格行为研究，会让项目从“上线准备”切回“策略探索”。

这会模糊主线目标。

### 2. “仅回测，不进 Paper Trading”价值有限

如果价格行为策略 Phase 5 只做回测、不进入 Paper Trading、不参与实盘，那它对当前主线的价值有限。

它能提供学习价值，但会占用最关键阶段的注意力。

更合适的位置是实盘主线稳定后，作为研究线并行推进。

### 3. TradingView 对比不适合作为验收基础

Claude 提到形态识别准确率可与 TradingView 对比。这一点我不建议作为硬标准。

原因：

- TradingView 不同指标脚本定义不一致
- 价格行为形态本身没有唯一标准
- 对比对象可能不可复现
- 容易把目标变成“拟合某个工具的标签”

如果未来做价格行为，验收应基于项目自定义规则是否一致、是否无前视偏差、是否可复现，而不是是否接近某个平台。

### 4. “学习价值”不应进入主线关键路径

我认同价格行为有学习价值，也认同它能帮助理解市场结构。

但主线目标应是构建一个可信交易系统，而不是最大化学习覆盖面。

学习线可以保留，但不应阻塞数据、回测、风控、Paper Trading 和小资金实盘。

   - 不进入 Phase 4 Paper Trading
   - 不进入 Phase 6 小资金实盘
   - 不影响实盘门禁

4. **验收不看收益**
   - 只看规则可复现
   - 只看无前视偏差
   - 只看数据和信号可解释

5. **AI 不参与标签定义**
   - AI 可以解释失败案例
   - 不能定义形态标准
   - 不能替代规则验收

如果这些限制都接受，方案 B 可以作为用户偏好下的可控选择。但我的推荐仍是 C+。

## 对验证标准的回应

### 数据质量检查

Claude Code 给出的数据质量清单基本完整，我建议补充三点：

1. **数据源冻结**
   - 每次回测必须记录数据下载时间、交易所、交易对、周期、起止时间。

2. **数据修复策略**
   - 缺口是重下、插值、还是标记跳过，必须有规则。
   - 默认不应静默插值。

3. **时区和 K 线闭合规则**
   - 明确所有时间统一为 UTC。
   - 明确策略只能使用已闭合 K 线。

### 回测验证标准

Claude Code 的标准方向正确，但我建议避免过早引入“蒙特卡洛 1000 次”作为硬门槛。

早期优先级应是：

1. 无前视偏差
2. 成本模型真实计入
3. 订单成交逻辑可解释
4. 数据版本可复现
5. 样本内/样本外划分
6. 参数敏感性检查

蒙特卡洛可以作为中期增强，不必成为早期阻塞项。

## 对 Paper Trading 的回应

60 天是合理的最低建议。

但 Paper Trading 不应只看“连续运行”，还应看偏差解释能力。

建议 Paper Trading 验收包括：

- 每个信号都有原因
- 每次不交易都有原因
- 每个订单状态可追溯
- 每日生成运行摘要
- 风控触发有记录
- 回测和模拟盘偏差可解释
- 系统暂停后需要人工恢复

我不建议要求“模拟盘结果与回测偏差 <30%”作为硬指标。市场状态不同，偏差未必能用固定百分比判断。

更好的要求是：偏差必须可解释。

## 对 Broker 架构的回应

认同三层架构：

- Paper Broker
- Exchange Broker
- Live Broker

但建议强调：早期不要急着实现完整 Exchange Broker。

Paper Broker 应尽量完善，重点模拟：

- 资金余额
- 仓位
- 手续费
- 滑点
- 限价单成交规则
- 订单取消
- 拒单
- 风控阻断

部分成交和订单簿深度可以作为中期增强，不必在第一版做复杂。

## 对“不交易”状态的回应

完全认同 NO_TRADE 和 PAUSE。

我建议再区分三个状态：

- `NO_SIGNAL`：没有交易信号
- `NO_TRADE`：有潜在信号，但环境不允许交易
- `PAUSE`：系统级暂停，需要人工或规则恢复

原因：

- `NO_SIGNAL` 是正常等待
- `NO_TRADE` 是策略拒绝
- `PAUSE` 是风险或系统保护

这三个含义不同，日志和复盘时必须区分。

## 对 AI 边界的回应

认同 Phase 7 前禁止 AI 自动执行。

我建议更严格：

```
Phase 1-6:
  AI 只读、只分析、只建议

Phase 7 小资金实盘:
  AI 仍然只建议，不自动执行

Phase 8+:
  可考虑半自动，但必须满足：
  - 参数变更有白名单
  - 每次变更有审计记录
  - 变更幅度有限制
  - 可一键回滚
  - 人工可关闭
```

不建议在早期承诺未来一定自动化。AI 自动执行应该是长期选项，不是默认目标。

## 最终立场

我不反对价格行为策略本身。

我反对的是：在数据、回测、Paper Trading、风控和小资金实盘尚未稳定之前，把价格行为放入主线关键路径。

建议现在就保留扩展设计，但不要启动价格行为实现。

先让系统证明它能：

- 使用可信数据
- 做可信回测
- 明确知道何时不交易
- 在模拟盘稳定运行
- 在小资金实盘中保护本金

之后再研究价格行为，会更稳。
