# 企划与开发流程优化建议

**目的：** 这份文档供 Claude Code 评审和补充。重点讨论项目方向、阶段设计、验证流程和风险控制，不讨论具体代码实现。

**当前判断：** 项目方向整体可行，但现在的企划更偏“功能驱动”。建议调整为“风险驱动 + 验证驱动”，先建立可信数据、可信回测、可信风控，再逐步进入策略扩展、AI 辅助和小资金实盘。

---

## 一、核心建议

### 1. 早期不要把 AI 优化作为主线

AI Agent 是项目亮点，但早期不建议让 AI 自动调参或参与自动交易决策。

早期 AI 更适合做：

- 回测报告解释
- 失败交易归因
- 风险清单检查
- 策略假设复盘
- 参数敏感性分析总结

不建议早期做：

- 自动应用 AI 参数建议
- AI 直接决定是否下单
- AI 自动切换策略
- AI 根据短期结果频繁优化参数

原因：早期数据量少，策略还未验证，自动优化很容易导致过拟合。

### 2. 数据质量优先于策略收益

Phase 1 不应只是“能下载数据”，而应是“能信任数据”。

数据阶段应明确回答：

- 数据是否有缺口
- K 线时间戳是否统一
- 是否存在重复 K 线
- 是否存在异常价格或异常成交量
- 数据源版本是否可追溯
- 回测使用的数据集是否可复现
- 不同周期数据是否由统一规则生成

如果数据不可审计，后续所有收益指标都不可靠。

### 3. 先做 Paper Trading 闭环，实盘接口后置

建议将交易执行概念拆成三层：

- Paper Broker：模拟撮合和账户状态
- Exchange Broker：交易所接口适配
- Live Broker：真实资金执行层

早期只需要 Paper Broker。

优先验证：

- 信号生成
- 模拟订单状态
- 仓位变化
- 手续费
- 滑点
- 风控熔断
- 暂停交易
- 交易日志和回放

实盘接口应是最后替换的一层，而不是早期主线。

### 4. 策略路线先收敛，不要过早多策略

当前文档路线是“网格 + 价格行为”。建议短期只保留一个主策略：

**网格策略 + 趋势过滤 + 波动率过滤 + 风控过滤**

价格行为策略可以保留为中期方向，但不建议过早推进。

原因：

- 价格行为主观性较强
- 验收标准难定义
- 很容易引入 AI 解释和过拟合
- 会拉长第一个可验证闭环

更稳的策略路线：

1. 纯网格策略
2. 网格 + 趋势过滤，避免强单边行情硬扛
3. 网格 + 波动率过滤，避免极端行情开仓
4. 网格 + 仓位管理
5. 长期再引入价格行为策略

### 5. 早期验收指标应从收益改为系统可信度

文档中已有年化收益、月收益、夏普比率等指标。这些可以作为观察指标，但不建议作为早期主要验收标准。

早期更重要的验收标准：

- 回测没有前视偏差
- 手续费和滑点真实计入
- 最大回撤计算正确
- 订单模拟逻辑合理
- 参数小幅变化不会导致结果剧烈反转
- 样本外结果不崩
- 模拟盘结果和回测偏差可解释
- 风控能正确暂停交易

收益指标应该在系统可信之后再作为主目标。

### 6. “不交易”应成为一等决策

交易系统不应永远寻找交易机会。建议把“不交易”设计成明确状态，而不是默认空白。

触发“不交易”的情况包括：

- 数据缺失或异常
- API 状态异常
- 波动率异常
- 趋势环境不适合网格
- 连续亏损达到阈值
- 当日亏损达到阈值
- 滑点或手续费过高
- 策略参数超出安全区间

这比增加新策略更重要。

## 二、建议的新阶段流程

### Phase 0：项目边界确认

目标：明确系统不做什么。

建议边界：

- 早期只做现货，不做合约
- 早期只做 BTC/USDT、ETH/USDT 等主流币
- 早期只做测试网和模拟盘
- 默认禁止 live trading
- AI 只做分析建议，不自动执行

验收：

- 风险边界写入文档
- 实盘前置条件明确
- 默认运行模式安全

### Phase 1：数据可信闭环

目标：建立可审计、可复现的数据基础。

核心问题：

- 数据从哪里来
- 数据是否完整
- 数据是否可复现
- 数据异常如何处理

验收：

- 能下载指定交易对历史数据
- 能检测时间缺口
- 能检测重复数据
- 能标记异常 K 线
- 能固定一次回测所使用的数据版本

### Phase 2：回测可信闭环

目标：验证回测引擎本身可信，而不是追求收益漂亮。

验收：

- 手续费计入
- 滑点计入
- 无前视偏差
- 最大回撤、收益率、胜率等指标定义清楚
- 能生成可复查报告
- 对同一数据和参数，回测结果可重复

### Phase 3：单策略验证

目标：先验证一个简单策略闭环。

建议策略：

- 网格策略
- 趋势过滤
- 波动率过滤
- 仓位控制
- 停止交易条件

验收：

- 策略能在不同市场环境下回测
- 参数敏感性可解释
- 样本外结果不过度崩溃
- 策略知道什么时候不交易

### Phase 4：Paper Trading

目标：验证模拟盘与回测之间的差异。

验收：

- 连续运行至少 30 天，建议 60 天
- 所有信号和订单可追溯
- 模拟账户权益曲线可复盘
- 实际模拟结果和回测偏差有解释
- 风控能正常暂停系统

### Phase 5：监控与风控强化

目标：先保护资金，再谈收益。

验收：

- 日亏损限制
- 最大仓位限制
- 连续亏损熔断
- 数据异常熔断
- API 异常熔断
- 人工恢复机制
- 报警机制

### Phase 6：AI 辅助分析

目标：让 AI 成为分析助手，而不是早期自动驾驶。

适合 AI 的任务：

- 每日报告总结
- 周度策略复盘
- 异常交易归因
- 参数敏感性总结
- 风险提醒

不建议早期开放：

- 自动调参后直接生效
- 自动切换策略
- 自动扩大仓位
- 自动进入实盘

### Phase 7：小资金实盘

目标：在前面闭环全部通过后，小资金验证真实环境。

前置条件：

- Paper Trading 至少 60 天
- 风控测试通过
- API Key 无提币权限
- 实盘开关需要显式开启
- 每次参数变更需要人工确认
- 初始资金建议低于 500 美元

## 三、建议修改的文档方向

### 1. 统一路线图

当前文档中存在几处口径不一致：

- 有的文档写趋势跟踪，有的写价格行为
- 有的写 30 天模拟盘，有的写 60 天
- 有的写 10 个 Skills，有的写 17 个 Skills

建议：

- 以 `GOALS_AND_DOCS.md` 和 `ROADMAP_UPDATE.md` 作为当前事实来源
- 旧完成总结类文档标注为历史记录
- 在 README 中只保留最新路线图摘要

### 2. 新增“实盘前门禁”文档

建议新增或规划一份：

`LIVE_TRADING_CHECKLIST.md`

内容包括：

- 实盘启用条件
- API Key 权限要求
- 风控参数要求
- 人工确认流程
- 回滚和紧急停止流程
- 禁止事项

### 3. 新增“策略假设文档”

建议新增或规划一份：

`STRATEGY_ASSUMPTIONS.md`

用于记录每个策略的核心假设：

- 适合什么市场
- 不适合什么市场
- 主要风险是什么
- 什么时候不交易
- 哪些参数不能自动调
- 如何判断策略失效

### 4. 新增“回测可信度标准”

建议新增或规划一份：

`BACKTEST_VALIDATION.md`

内容包括：

- 前视偏差检查
- 手续费和滑点模型
- 数据版本记录
- 样本内/样本外划分
- Walk-forward 原则
- 参数敏感性测试

## 五、最终建议

当前项目不缺功能规划，缺的是更严格的验证顺序。

建议将项目定位从：

> 构建一个完整的加密货币自动交易系统

调整为：

> 构建一个数据可信、回测可信、风控优先、逐步验证的加密货币交易系统

早期少做策略，多做验证；少追收益，多查错误；少自动化执行，多人工确认。

这会降低开发速度，但能显著降低进入实盘后的系统性风险。
