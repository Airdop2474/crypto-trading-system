# 优化方案 v3：基于 v2 的多角色再辩论

## 背景

v2 在"现实校准"上做得不错（12 策略 + VPS + 实时纸盘），但仍有几个未充分讨论的问题：

1. 止损是否会反而打掉现有策略的 edge？
2. Telegram 通知是否该绑定 Phase 1？
3. 5 周工期的依据是"砍 47%"，是否真实可执行？
4. 技术债"每 Phase 10%"是否是空话？
5. Phase 4 的杂物组合（成交量 + 日报 + 前端）是否值得？

本文用 4 个新角度重审 v2，输出 v3。

## 参与角色

- **Sk（怀疑者）**：拆解假设，挑战"应该"和"会"
- **Tr（交易员）**：钱第一，关心系统什么时候真正开始保护/产生收益
- **Eng（工程师）**：现实工期，知道 1 周任务通常等于多少周
- **Stat（统计学家）**：方法论守门，反对 p-hacking 和未验证的数字
- **Arch（架构师）**：综合输出 v3

---

## 第一轮：止损会不会反而打掉现有策略的 edge？

### Sk

> v2 默认"加止损 = 更安全"。但 12 策略的 backtest 都是**无止损版**。加 ATR 止损不是单纯"加上限保护"，而是**修改了策略本身的 P&L 分布**——这是策略变更，不是安全包装。

### Stat

> 以 RSI 为例。RSI 是均值回归策略，很多 winning trade 入场后会经历 -2% 到 -4% 浮亏后反弹。ATR 1.5x 止损（典型 ≈ -2.5%）会把这些 winning trade 提前砍掉。可能的结果：最大回撤下降 30%，但年化收益**下降 50%**。"安全"了，但更不赚钱。
>
> v2 的 acceptance criteria "RSI 最大回撤下降 > 20%" 也偏弱——没说时间窗口、基线、置信区间。这种条件评估时会被默认通过。

### Tr

> 那不就是为了安全感把策略毁了？

### Eng

> v2 的设计是 Phase 1 加止损，Phase 2 才评估。意味着两周里实盘会跑**没有验证过的"加了止损"版本**。这个 gap 不能接受。

### 共识（强制）

Phase 1 出口标准必须含每个策略**加止损前后的对照回测**：

- 回测窗口 ≥ 12 个月（含牛/熊周期）。
- Sharpe 下降 > 30% → 该策略止损方案返工或暂时禁用实盘。
- 最大回撤下降 < 15% → 止损未起效，重检查参数。

---

## 第二轮：Telegram 应该和止损一起做吗？

### Sk

> v2 把 Telegram 放 Phase 1，理由是"止损触发要通知"。但这是**绑定**的理由吗？Telegram 是通信通道，止损是风险管理，关注点完全不同。

### Eng

> Phase 1 标 2 周。光止损（基类 + 2 类止损 + 12 策略集成 + 重验证）就够 3 周。再加 Telegram（异步 + 3 级 + 静默 + 降级 + 集成）1 周。塞一起 = 2 周根本不够。

### Tr

> 止损没上线之前，Telegram 通知什么？"我们还没保护你的钱"？Telegram 真正有价值的时机是 **Day 0 的 daemon 看门狗**——daemon 挂了我要立刻知道。这个独立于止损。

### Sk

> v2 的 Phase 0 已经写了"daemon 看门狗 → Telegram"。Telegram 实际上是 **Phase 0 的依赖**，不是 Phase 1 的产物。

### 共识

Telegram 提前到 Day 0，与 watchdog 一起实施。Phase 1 只关注止损 + 重验证。

---

## 第三轮：5 周工期是否现实？

### Eng

> 用过去类似项目对比工期：
>
> | v2 Phase | 标定 | 实际估计 | 差距原因 |
> |---|---|---|---|
> | Phase 1（止损 + Telegram） | 2 周 | 3 周 | 12 策略集成 + 重验证不可省 |
> | Phase 2（MC + 评估 + 前端） | 1 周 | 2 周 | MC 框架 + 12 策略评估 + 报告生成 |
> | Phase 3（Heat + 技术债） | 1 周 | 1 周 | 这个是合理的 |
> | Phase 4（成交量 + 日报 + 前端） | 1 周 | 任意一项接近 1 周 |
>
> 5 周是从 v1 的 13.5 周砍 47% 得来——这是结果不是论证。

### Stat

> v2 多处 acceptance criteria 不可量化："RSI 最大回撤下降 > 20%"——什么时段？哪种基线？没说。"淘汰建议合理（与人工判断一致率 > 80%）"——这是循环论证（人工判断本身就不严格）。

### Sk

> 5 周的真正问题是：止损这种 high-stakes 改造**没有给重验证留时间**。这是计划本身的缺陷。

### 共识

v3 工期改为 **6 周**（确定）+ 0-1 周（条件 Phase 4）。

| Phase | v2 工期 | v3 工期 |
|---|---|---|
| Day 0 | 1 天 | 1 天（含 Telegram） |
| Phase 1 | 2 周 | **3 周**（含重验证） |
| Phase 2 | 1 周 | **2 周**（含淘汰规则白皮书） |
| Phase 3 | 1 周 | 1 周 |
| Phase 4 | 1 周 | **0-1 周**（条件触发） |

---

## 第四轮：技术债"每 Phase 10%"可执行吗？

### Eng

> 我做过 5 年项目，没见过"每 Phase 留 10% 处理技术债"真正落实过。Phase 主任务永远超时，10% 第一个被牺牲。这是空话。

### Arch

> v2 把双轨制清理放 Phase 3 是对的——因为 Portfolio Heat 要写 state 文件，必须先统一数据源。这是**依赖驱动**的清理，不是"挤进 10%"。

### 共识

- 保留 Phase 3 = Portfolio Heat + 双轨制清理（依赖驱动，必须）。
- 删除其他 Phase 的"10% 技术债"约定。
- 技术债视作**一次性集中清理**，不是持续性活动。

---

## 第五轮：Phase 4 真的值得吗？

### Tr

> 5-6 周后系统更稳更安全，但**收益没变**。Phase 4 是唯一带 alpha 增强潜力的，所以应该做。

### Stat

> Donchian 成交量确认"假突破率下降 > 15%" 这个数字**没数据支撑**。需要先用 Phase 2 的 MC 评估 Donchian 实际表现，才能定量讨论是否需要、需要哪种增强。

### Sk

> 模板日报和前端优化和"赚钱"几乎无关。和成交量增强混在一起会让 Phase 4 变成杂物 Phase。

### 共识

- 成交量增强**依赖 Phase 2 评估结果**：Donchian 表现差先调 Donchian 本身；表现好再加成交量确认验证是否进一步提升。
- 模板日报降级 P2，不在 6 周路线图。
- 前端优化作为后台任务单独跟踪。

---

## 第六轮：v2 缺失的关键环节

### Stat

> Phase 2 写"输出评分和淘汰建议"——但**淘汰依据是什么**？没标准就是主观。我建议明确 4 阈值：
>
> - Sharpe Ratio < 0.3 → 淘汰候选
> - 最大回撤 > 25% → 淘汰候选
> - 与其他策略 6 月相关性 > 0.85 → 冗余候选
> - 样本外 vs 样本内 Sharpe 差异 > 50% → 过拟合候选
> - 命中 ≥ 2 项 → 列入淘汰

### Tr

> 还有一个：**回滚计划**。Phase 1 加止损后如果整体表现变差，怎么回滚？v2 没说。

### Eng

> 建议每个 Phase 开始前打 git tag + 数据库备份点，方便定位回滚目标。

### Sk

> 实盘切换计划这事 v2 完全没提，但用户大概率在 v4 会问。先记下，不放进 v3 范围。

### 共识

- Phase 2 出口**必须**含淘汰规则白皮书（4 阈值 + 组合判定）。
- 每个 Phase 开始前打 git tag + 备份。
- 实盘切换计划 = v4 议题。

---

## v3 方案：6 周路线图

### 总览

```
Day 0   : 运维加固 + Telegram + Watchdog
Week 1-3: Phase 1 - 止损体系 + 12 策略对照重验证
Week 4-5: Phase 2 - Monte Carlo + 策略评估 + 淘汰规则白皮书
Week 6  : Phase 3 - Portfolio Heat + 双轨制清理
TBD     : Phase 4 - 条件 Phase（按 Phase 2 评估结果决定）
```

总工期：**6 周**（确定）+ 0-1 周（条件触发）。

### Phase 0：运维加固 + Telegram（Day 0，1 天）

| 任务 | 说明 | 优先级 |
|------|------|--------|
| Docker 日志轮转 | max-size + max-file | P0 |
| 数据库备份 | 每日 pg_dump + 7 天保留 | P0 |
| 一键更新脚本 | git pull + docker compose up -d --build | P0 |
| **TelegramNotifier** | 异步，3 级（CRITICAL/WARNING/INFO），无 token 时降级日志 | **P0（提前）** |
| daemon 看门狗 | cron 每 5 分钟检查 + Telegram CRITICAL | P0 |
| 磁盘监控 | > 85% Telegram WARNING | P0 |
| 启动 git tag | `v3-phase0-start` | P0 |

**出口标准**：
- Telegram 收到测试消息（< 5 秒）。
- 故意杀死 daemon → 5 分钟内收到告警。
- 备份脚本可回放到上次备份。
- 无 Token 配置时所有功能正常运行（仅日志输出）。

### Phase 1：止损体系 + 重验证（Week 1-3，3 周）

| 任务 | 说明 | 优先级 |
|------|------|--------|
| StopLossManager | 统一门面（ATR/移动/区间/时间） | P0 |
| RiskAwareStrategy 集成 | 基类自动调用，子类只需 entry_price/entry_time | P0 |
| 趋势策略止损 | ATR 1.5x + 移动止损（激活 3%、回撤 3%） | P0 |
| 均值回归止损 | 区间突破 5%（Grid/BB/Reversal/PriceAction） | P0 |
| 时间止损 | 50 K 线未达标平仓 | P1 |
| 单元测试 | 每种止损 ≥ 5 用例 | P0 |
| **12 策略对照回测** | 加止损前/后并排，窗口 ≥ 12 个月 | **P0（出口标准）** |
| 通知集成 | 止损触发 → Telegram WARNING | P0 |
| 静默模式 | 用户配置接收级别 | P1 |

**出口标准**：
- 11 策略（除 BuyHold）全部有止损。
- 对照回测：每策略 Sharpe 下降 < 30%，最大回撤下降 > 15%。
- 任何策略不达标 → 该策略止损方案返工或实盘暂时禁用。
- Telegram 在测试中触发 ≥ 1 条止损通知。
- 现有测试套件全绿。
- git tag `v3-phase1-end`。

### Phase 2：Monte Carlo + 策略评估 + 淘汰规则（Week 4-5，2 周）

| 任务 | 说明 | 优先级 |
|------|------|--------|
| MonteCarloSimulator | NumPy 向量化，1000 模拟 < 5 秒 | P0 |
| 参数稳定性 | ±10% 微扰灵敏度 | P0 |
| 样本外验证 | 70/30 自动分割 | P0 |
| **淘汰规则白皮书** | Sharpe<0.3 / MDD>25% / Corr>0.85 / IS-OS>50% 四阈值 | **P0** |
| 12 策略评估报告 | 输出 score + 淘汰建议 + 数据支撑 | P0 |
| 结果存储 | 摘要存 DB（不存明细），≤ 1KB/策略 | P0 |
| 前端展示 | 分析中心 MC 页（可推迟） | P1 |

**出口标准**：
- 12 策略全有评估报告（median P&L / VaR / CVaR / 参数灵敏度 / IS-OS 差异）。
- 淘汰规则白皮书发布并 review。
- 至少 1 个淘汰建议执行（实盘禁用或参数调整）。
- VPS 上 1000 模拟实测 < 5 秒。
- git tag `v3-phase2-end`。

### Phase 3：Portfolio Heat + 双轨制清理（Week 6，1 周）

| 任务 | 说明 | 优先级 |
|------|------|--------|
| Portfolio Heat | daemon 计算组合热力，写 state | P0 |
| 超阈值拒单 | 组合热力 > 15% 拒绝新开仓 + Telegram WARNING | P0 |
| 双轨制清理 | 删除 service.py 的 `_build_state()` 预跑逻辑 | P0 |
| API 端点统一 | 全部优先 live_data，预跑数据仅 fallback | P0 |
| service.py 拆分 | backtest/analytics/strategy 三模块 | P1 |
| 前端展示 | 仪表盘加组合热力指标 | P1 |

**出口标准**：
- 组合热力 > 15% 时实测拒单生效。
- service.py 中预跑逻辑删除后所有 API 测试通过。
- 现有测试套件全绿。
- git tag `v3-phase3-end`。

### Phase 4：条件 Phase（按 Phase 2 结果，0-1 周）

仅在以下条件触发：

- **Donchian 成交量确认**：Phase 2 评估显示 Donchian 假突破率 > 15%（实证支撑）→ 加成交量确认。
- **MTF 工具函数**：Phase 2 显示某策略受高时间框架趋势影响显著 → 加 `get_high_tf_trend()` 工具函数。

否则 Phase 4 跳过。

**模板日报、恐惧贪婪指数、前端优化** 作为后台任务单独跟踪，不计入主路线图。

## v3 vs v2 差异表

| 维度 | v2 | v3 | 原因 |
|------|----|----|------|
| 总工期 | 5 周 | 6 周（确定）+ 0-1 周 | 重验证不能省 |
| Telegram 时机 | Phase 1 | **Day 0** | watchdog 依赖 Telegram，本身就该提前 |
| Phase 1 范围 | 止损 + Telegram | 止损 + **对照重验证** | 止损会改 P&L 分布，必须验证 |
| Phase 2 工期 | 1 周 | 2 周 | MC + 12 策略评估 + 白皮书 |
| 淘汰规则 | "建议" | **白皮书 + 4 阈值** | 主观无法落地 |
| Phase 4 | 必做 | **条件触发** | 缺数据支撑 |
| 技术债 | 每 Phase 10% | 仅 Phase 3 | "10%"不可执行 |
| 回滚 | 无 | **每 Phase git tag + 备份** | 加止损可能变差 |
| 出口标准 | 模糊 | 量化 + 时间窗口 + 不达标处置 | 防止默认通过 |

## 关键风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 止损打掉策略 edge，Sharpe 下降 > 30% | 中 | 高 | Phase 1 出口对照回测，不达标返工或禁用 |
| Phase 2 MC 在 VPS 上跑超时 | 低 | 中 | 本地跑后上传摘要；NumPy 向量化 |
| 双轨制清理破坏现有 API | 中 | 高 | Phase 3 前完整 API 集成测试 |
| 6 周仍超时 | 中 | 中 | 每 Phase git tag，超时 1 周内强制评审 |
| Telegram API 限流 | 低 | 低 | 异步队列 + 退避重试 |
| 12 策略对照回测时间不够 | 中 | 中 | 并行回测 + 优先关键策略（RSI/MA/Donchian/SuperTrend） |

---

## v3 没解决的（留给 v4）

- 实盘切换计划（Testnet → 真盘）
- 多账户/多交易所支持
- 增强滑点 / 资金费率建模
- 长期 alpha 来源研究（链上、宏观）
- LLM 日报（看 Phase 2 数据再定）

---

## 关键设计决策（v3 新增/调整）

| 决策 | 理由 |
|------|------|
| Telegram 提前到 Day 0 | watchdog 依赖；与止损解耦 |
| Phase 1 含对照重验证 | 止损会改 P&L 分布，不验证就上线是赌博 |
| Phase 2 含淘汰白皮书 | 没量化阈值的"淘汰建议"= 主观 |
| Phase 4 条件触发 | 没数据支撑就别预设 |
| 删除"每 Phase 10% 技术债" | 实践证明不可执行 |
| 每 Phase git tag + 备份 | 高风险变更必须可回滚 |
| 6 周而非 5 周 | 砍工期不该砍验证 |
