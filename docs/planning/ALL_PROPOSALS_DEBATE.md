## 第一轮：整体优先级之争

### QS（量化策略师）

> 综合方案把止损放 Phase 1、新策略放 Phase 5，我有异议。理由：
>
> 1. **新策略才是 alpha 来源**。止损只能减少亏损，但不会创造收益。没有好的策略，再好的风控也只是"少亏"。
> 2. 当前只有 8 个策略，且趋势策略占 5/8，策略类型严重失衡。**BB 均值回归**和 **VWAP** 是填补空白的刚需。
> 3. MTF 过滤器能直接提升现有 8 个策略的胜率 5%+，这是"一对多"的杠杆效应。
>
> 我建议：Phase 1 应该是"MTF 过滤器 + BB 策略"，止损放 Phase 2。

### RE（风控工程师）

> **强烈反对**。QS 你的思路是"先造发动机，再装刹车"。这在交易系统里是致命的。
>
> 1. 当前系统**没有任何交易级止损**。RiskAwareStrategy 的熔断是"策略级"的——连亏 5 笔才触发，日亏 5% 才暂停。但单笔交易可能已经亏了 10% 才被熔断兜住。
> 2. 现实场景：BTC 在 4h 级别单根 K 线可以跌 8%。没有止损的话，一笔交易就能吃掉大半年的利润。
> 3. **止损是基础设施，不是功能**。所有策略都应该先有止损，再谈收益。
>
> 我的立场：**Phase 1 必须是止损体系**，没有任何讨价还价的余地。

### SA（系统架构师）

> 我支持 RE 的优先级排序，但我要补充一个技术视角：
>
> 1. 方案 B 的止损模块是**独立的工具类**（`ATRStopLoss`, `TrailingStop`, `TimeStop`），不影响现有策略的 `on_bar()` 逻辑。策略只需要在 `on_bar()` 开头加一个止损检查。
> 2. 方案 A 的 MTF 过滤器需要**架构改动**——`MultiStrategyRunner` 的 `StrategyConfig` 要新增 `high_tf_data` 字段，`BacktestEngine` 可能也需要适配。
> 3. **先做简单改动（止损），再做架构改动（MTF）**，这是工程上的正确顺序。
>
> 另外我要指出：综合方案的 Phase 1 把 Telegram 和止损放在一起，这是**合理的打包**。Telegram 是独立模块，不会阻塞止损的交付，两者可以并行开发。

### PM（产品经理）

> 从交付角度看：
>
> 1. **止损 + Telegram 放 Phase 1 是合理的**，因为这两个都是"实盘必需品"。没有止损不能上线，没有通知就无法监控实盘。
> 2. 但 15 周的总工期太长了。我们的用户需要更快看到价值。
> 3. 我建议砍掉 Phase 7 的大部分内容。**PortfolioManager 和前端对比页面可以延后**，先用命令行和 Grafana 看结果。
>
> 我的 MVP 建议：Phase 1（止损+Telegram）+ Phase 2（Monte Carlo）= 3.5 周就能上线一个更安全的系统。

## 第二轮：方案 A 内部辩论

### 新策略的必要性

**QS**：BB、VWAP、Momentum 三个策略填补了均值回归和成交量分析的空白。特别是 BB 策略，在震荡市中可以替代 Grid 的固定区间，自适应性更强。

**RE**：我反对一次性开发 3 个新策略。原因：
1. 当前 8 个策略还没有经过实盘验证。在没确认"现有策略能赚钱"之前，新增策略是**扩大未知领域**。
2. 新策略需要 Monte Carlo + 参数稳定性验证（方案 C Phase 2）。如果先开发策略再做验证，等于**先生产再质检**。
3. 建议：**先开发 BB 策略一个**，用它来验证整个"开发-验证-上线"流程，确认流程跑通后再开发其他。

**SA**：同意 RE。而且我要补充：BB 策略的实现最简单（只需要 rolling mean + rolling std），适合作为"流程验证器"。

**QS**：（思考后）接受只做 BB 一个作为先导。但 VWAP 策略的成交量分析能力必须通过"成交量增强"模块补上——也就是在 Donchian 和 RSI 上增加成交量确认。这是低成本高收益的改动。

**PM**：同意。成交量增强是改动现有代码，不是新建文件，工期短、风险低。

**共识**：Phase 5 只做 BB 策略 + 成交量增强，VWAP 和 Momentum 延后。

### MTF 过滤器

**QS**：MTF 过滤器是方案 A 中价值最高的模块。它是一个**装饰器**，可以套在任何策略上，一对多提升胜率。

**SA**：MTF 的设计我有顾虑：
1. 需要高时间框架数据（1d），但当前系统只存 4h 数据。**数据管道要改**。
2. `MTFWrapper` 对外是 Strategy，但内部需要访问两个时间框架的数据。当前 `on_bar()` 的签名只接收一个 DataFrame。
3. 建议：MTF 改为**策略内部调用工具函数**，不做成包装器。每个策略在 `on_bar()` 中自己调用工具函数获取高 TF 趋势并判断。

**QS**：这样每个策略都要重复调用逻辑，违反 DRY 原则。

**SA**：可以提供一个**工具函数** `get_high_tf_trend(data, current_time, ema_period)` 供策略调用。策略内部调用工具函数，不重复逻辑，也不需要改架构。

**PM**：这样更好。避免了架构改动，降低了 Phase 5 的风险。

**共识**：MTF 用工具函数实现，不改 MultiStrategyRunner 架构。

### 自适应参数管理器

**QS**：AdaptiveParamManager 让策略根据 MarketClassifier 的输出自动切换参数。这是"短期自适应"，和 EvolutionEngine 的"长期优化"互补。

**RE**：我对"实时切换参数"持保留态度。理由：
1. MarketClassifier 的分类不是 100% 准确的。如果误判市场状态，会切换到错误的参数。
2. 频繁切换参数会导致策略行为不稳定，回测和实盘不一致。
3. 建议：增加**最小切换间隔**（如 72 小时），并且切换后有一个"观察期"（如 4 小时内不交易）。

**SA**：同意 RE。而且自适应参数管理器应该**不直接修改策略实例的参数**，而是返回一个"建议参数"对象，策略可以选择是否采纳。这样更安全。

**PM**：这个模块的优先级应该降低。理由：现有 8 个策略都用静态参数跑，还没证明静态参数不行。先跑一段时间实盘，如果发现"市场状态切换时策略表现明显变差"，再加自适应。

**共识**：自适应参数管理器降级为 P3，延后到 Phase 6。

### PortfolioManager

**QS**：PortfolioManager 是方案 A 的顶层设计，负责策略间资金分配和相关性监控。没有它，多策略并行只是"各自为战"。

**SA**：PortfolioManager 的设计过于复杂。它同时做资金分配、相关性监控、组合级风险控制三件事。建议拆分：
1. 资金分配 -> `PortfolioAllocator`（简单版：等权或风险平价）
2. 相关性监控 -> 方案 B 的 `CorrelationMonitor`（已存在）
3. 组合级风控 -> 扩展 `RiskManager`（方案 B 已有 `calculate_portfolio_heat`）

**RE**：同意 SA 的拆分。PortfolioManager 不应该是一个"上帝对象"。

**PM**：而且 PortfolioManager 的前置条件是"有 3 个以上稳定的策略"。现在连止损都没有，谈组合管理为时过早。

**共识**：PortfolioManager 拆分为三个独立模块，延后到 Phase 7。先用方案 B 的 Portfolio Heat 和 Correlation Monitor 代替。

## 第三轮：方案 B 内部辩论

### 止损体系的细节

**RE**：方案 B 的三层止损设计（ATR 初始止损 + 移动止损 + 时间止损）是合理的。但我要强调几个细节：

1. **ATR 止损的 min/max 钳位**：`min_stop_pct=0.01`（1%）和 `max_stop_pct=0.10`（10%）是合理的。但 BTC 在高波动时 4h K 线的 ATR 可能达到 3-4%，2 倍 ATR 就是 6-8%。这意味着在高波动时止损会放宽到 6-8%，对小资金账户来说单笔亏损过大。
2. **建议**：对小资金账户（< $2000），`max_stop_pct` 应该降到 5%。或者用**账户风险百分比**来反推止损距离。

**QS**：同意 RE。止损应该和仓位大小联动。如果 Kelly 仓位算出来只有 5% 仓位，止损可以放宽一些（因为绝对亏损小）。如果仓位是 25%，止损必须收紧。

**SA**：这就是为什么止损模块和仓位管理模块应该**一起设计**，虽然分两个 Phase 实施。接口要预留仓位参数。

**PM**：同意。先实施止损（Phase 1），但接口设计要考虑到 Phase 3 的 Kelly 仓位。

### 移动止损的激活阈值

**RE**：`activation_profit=0.02`（2%）太低了。在 4h 级别，BTC 很容易在一根 K 线内波动 2%，这意味着移动止损几乎立刻激活。建议提高到 5%。

**QS**：5% 太高了。如果 BTC 涨了 5% 再回撤 3%，移动止损才触发，实际只赚了 2%。还不如不用移动止损，直接用 ATR 止损。

**RE**：那 3% 吧。涨 3% 激活，从最高点回撤 3% 触发。这样至少保证不亏损。

**PM**：这个参数应该做成可配置的，让每个策略自己决定。不同策略的风险偏好不同。

**共识**：`activation_profit` 默认 3%，可配置。`trail_pct` 默认 3%，可配置。

### Kelly 仓位的可靠性

**QS**：Kelly 公式需要"最近 30 笔交易"的数据。但新策略上线初期没有 30 笔交易怎么办？

**RE**：方案 B 已经处理了——数据不足时用 `min_position`（5%）。但我担心另一个问题：**Kelly 公式对胜率和赔率的估计非常敏感**。如果最近 30 笔交易恰好集中在某种市场状态，Kelly 会给出极端的仓位建议。

**SA**：建议用**半 Kelly**（Kelly/2）并且加上额外的**移动平均平滑**。不要用最近 30 笔的点估计，用最近 100 笔的移动平均。

**PM**：100 笔太多了，新策略可能需要跑几个月才有 100 笔。60 笔吧。

**共识**：`lookback_trades` 默认 60，用半 Kelly，增加 5 笔移动平均平滑。

### 滑点建模

**SA**：方案 B 的增强滑点模型需要 `avg_daily_volume` 参数。但当前系统没有存储日均成交量。需要先在数据层增加这个统计量。

**RE**：而且回测时用的是历史成交量，实盘时用的是实时成交量。两者可能差异很大。

**SA**：建议：回测时用滚动 20 日日均成交量，实盘时用最近 24 小时成交量。两者都从数据库读取。

**共识**：滑点模型先用简单版（固定滑点 + 订单大小系数），不做复杂的成交量建模。

## 第四轮：方案 C 内部辩论

### Monte Carlo vs 参数稳定性

**SA**：方案 C 把 Monte Carlo 和参数稳定性分成两个模块。但它们的**输入数据相同**（都是交易记录），**目的相同**（都是评估策略可信度）。建议合并为一个 `StrategyValidator` 模块。

**RE**：不同意合并。Monte Carlo 评估的是"交易序列风险"（最差情况有多差），参数稳定性评估的是"参数敏感性"（参数微扰后结果变化多大）。两者是独立的维度。

**QS**：我支持 RE。一个策略可以 Monte Carlo 表现很好（交易序列稳定），但参数稳定性很差（对参数微扰敏感）。两者都需要独立评估。

**SA**：（退让）好吧，保持独立。但它们应该共享一个"验证结果"数据结构，方便前端统一展示。

**共识**：保持独立模块，但共享 `ValidationResult` 数据结构。参数稳定性作为 Monte Carlo 模块的扩展方法实现，不增加工期。

### LLM 的定位

**PM**：LLM 市场日报和交易日志是方案 C 中用户感知最强的功能。每天收到一份中文日报，看到每笔交易都有解释，用户体验直接拉满。

**SA**：我反对把 LLM 作为核心功能。理由：
1. **成本不确定**：GPT-4 每次调用约 $0.01-0.05，每天生成日报 + 每笔交易解释，一个月可能 $10-30。对小资金账户来说是额外负担。
2. **可靠性问题**：LLM 输出不可预测。可能生成不准确甚至误导性的分析。
3. **依赖外部服务**：网络问题、API 限流、模型更新都可能影响功能。

**PM**：那用本地 LLM 呢？

**SA**：本地 LLM 的推理质量取决于模型大小。7B 模型的金融分析能力有限，13B+ 需要 GPU，又增加硬件成本。

**RE**：我建议分层：
1. **交易日志**：用模板生成，不需要 LLM。模板已经能覆盖 90% 的场景。
2. **市场日报**：LLM 作为可选增强。默认用模板，有 API Key 时用 LLM。

**QS**：同意 RE。而且日报的**核心价值是数据整合**，不是自然语言生成。把当日行情、策略表现、风险指标汇总成结构化数据，用模板渲染就已经很有用了。

**PM**：（退让）好吧。日报先用模板，LLM 作为 Phase 6 的可选增强。

**共识**：
- 交易日志：纯模板，不依赖 LLM
- 市场日报：Phase 5 用模板版，Phase 6 可选升级为 LLM 版
- LLM 成本：月预算上限 $20，超出自动降级到模板

### Telegram vs 健康度评分

**PM**：Telegram 通知是实盘必需品，必须尽早做。健康度评分是锦上添花，可以延后。

**RE**：同意。Telegram 的优先级远高于健康度评分。理由：
1. 熔断触发时需要立刻通知，否则用户不知道系统已暂停。
2. 异常交易（如滑点过大、API 失败）需要实时告警。
3. 健康度评分可以每天在日报中展示，不需要独立模块。

**SA**：Telegram 的实现很简单（一个 HTTP POST），可以和止损一起做。健康度评分的评分维度设计需要更多讨论，延后合理。

**共识**：
- Telegram：Phase 1（和止损一起）
- 健康度评分：合并到日报中，不做独立模块

### 链上数据的优先级

**QS**：链上数据（交易所净流入、巨鲸地址）是独特的信号源，能提供传统技术分析看不到的信息。

**RE**：链上数据的**信号质量存疑**。交易所净流入和价格的关系不是线性的。大量 BTC 流入交易所可能是为了卖，也可能是为了参与 Launchpad。

**SA**：而且 CryptoQuant 的免费 tier 有调用限制。如果数据获取失败，需要降级策略。这增加了系统复杂度。

**PM**：链上数据的投入产出比最低。建议放到 Phase 4 的可选部分。先用免费的恐惧贪婪指数和资金费率，这两个数据源稳定且免费。

**共识**：链上数据降级为 P3（可选），Phase 4 先做恐惧贪婪指数和资金费率。

### 组合优化器 vs PortfolioManager

**SA**：方案 C 的 `PortfolioOptimizer` 和方案 A 的 `PortfolioManager` 有功能重叠。两者都涉及资金分配。

**QS**：区别在于：PortfolioOptimizer 是离线工具（在回测中计算最优权重），PortfolioManager 是在线组件（实时调整资金分配）。

**SA**：建议：PortfolioOptimizer 作为回测工具先行，输出的权重作为 PortfolioManager 的输入。但 PortfolioManager 延后实施，先用等权分配。

**PM**：同意。等权分配 + 方案 B 的 Portfolio Heat 已经足够初期使用。

**共识**：
- PortfolioOptimizer：合并到 Phase 2（回测工具，和 Monte Carlo 一起）
- PortfolioManager：Phase 7（在线组件），初期用等权分配

## 第五轮：综合方案的调整

### PM 提出的 MVP 时间线

**PM**：经过四轮讨论，我重新整理一个 MVP 时间线：

| Phase | 内容 | 工期 | 交付物 |
|-------|------|------|--------|
| 1 | 止损体系 + Telegram | 2 周 | stop_loss.py, telegram_notifier.py, 策略集成 |
| 2 | Monte Carlo + 参数灵敏度 + 增强滑点 | 1.5 周 | monte_carlo.py, 滑点改进 |
| 3 | Kelly 仓位 + Portfolio Heat | 1.5 周 | position_sizer.py, risk_manager 增强 |
| 4 | 恐惧贪婪 + 资金费率 | 1 周 | sentiment.py, exchange.py 扩展 |
| 5 | BB 策略 + 成交量增强 + MTF 工具 + 模板日报 | 2 周 | bollinger_band.py, 策略改动, daily_report.py |
| **总计** | | **8 周** | |

**砍掉的内容**：
- PortfolioManager（用等权 + Portfolio Heat 代替）
- 健康度评分（合并到日报）
- 链上数据（可选，用户自行决定）
- LLM 日报（可选升级）
- 自适应参数（延后）
- VWAP / Momentum 策略（延后）
- 参数稳定性独立模块（合并到 Monte Carlo 扩展方法）
- 随机入场测试（延后）

### SA 的架构审查

**SA**：对 PM 的时间线，我有几个架构层面的意见：

1. **止损模块的接口设计**：`ATRStopLoss`, `TrailingStop`, `TimeStop` 应该统一为一个 `StopLossManager` 门面类，策略只需要和一个对象交互。内部组合三种止损逻辑。

2. **Telegram 的异步设计**：通知必须异步发送，不能阻塞交易主循环。用 `asyncio.create_task()` 或独立的发送队列。

3. **Monte Carlo 的性能**：1000 次模拟在纯 Python 中可能需要 10-30 秒。建议用 NumPy 向量化，目标 5 秒内完成。

4. **数据层扩展**：恐惧贪婪指数和资金费率应该有**统一的数据注册机制**，类似策略的 Registry。新数据源注册后自动被 MarketClassifier 使用。

**RE**：补充一点——所有新模块必须有**降级策略**。如果 Telegram 发送失败，日志记录失败原因但不阻塞交易。如果外部数据获取失败，回退到纯 OHLCV。

### QS 的最后争取

**QS**：我接受大部分调整，但要为**参数稳定性分析**争取一下。理由：

Monte Carlo 和参数稳定性评估的是**不同的风险维度**：
- Monte Carlo：交易序列的随机性 -> "如果运气差，会亏多少？"
- 参数稳定性：参数选择的敏感性 -> "如果参数选错了，会亏多少？"

一个策略可以 Monte Carlo 通过（交易序列稳定），但参数稳定性不通过（对参数微扰敏感 = 过拟合）。两者互补。

**RE**：我同意 QS。参数稳定性分析可以作为 Monte Carlo 的**扩展**实现，不需要独立模块。在 `monte_carlo.py` 中增加一个 `parameter_sensitivity()` 方法。

**SA**：这样实现成本很低，同意。

**共识**：参数稳定性作为 Monte Carlo 模块的扩展方法，不增加工期。

## 辩论总结

### 共识达成的关键点

1. **风控先行**：所有角色一致同意止损体系是最高优先级。没有止损的交易系统不是交易系统，是赌博。

2. **MVP 思维**：从 15 周压缩到 8 周，砍掉 47% 的工作量。核心原则：先做"必须有"（止损、通知、Monte Carlo），再做"应该有"（新策略、数据扩展），最后做"锦上添花"（LLM、组合管理）。

3. **独立模块优先**：避免架构改动。MTF 用工具函数而非包装器，止损用独立类而非修改策略基类，Telegram 用异步队列而非修改主循环。

4. **降级策略**：所有外部依赖（Telegram、API、LLM）都有降级方案。系统在完全离线状态下也能正常运行。

5. **验证驱动开发**：Monte Carlo + 参数稳定性在 Phase 2 完成，后续所有新策略（Phase 5）必须通过验证才能上线。

### 被砍掉/延后的功能清单

| 功能 | 原方案 | 处理方式 | 延后条件 |
|------|--------|---------|---------|

| Momentum 策略 | A Phase 1 | 延后 | BB 策略验证通过后 |
| 自适应参数 | A Phase 2 | 延后 | 发现市场状态切换影响收益后 |
| PortfolioManager | A Phase 3 | 拆分延后 | 3+ 策略稳定运行后 |
| 链上数据 | C Phase 1 | 降级为可选 | 有明确的 alpha 来源后 |
| LLM 日报 | C Phase 2 | 降级为可选 | 月交易盈利 > $50 后 |
| 健康度评分 | C Phase 3 | 合并到日报 | - |
| 随机入场测试 | C Phase 2 | 延后 | Monte Carlo 稳定后 |
